Introduction

eMule AI modernizes the eMule experience with NAT Traversal, QUIC/uTP and IPv6 support, stronger LowID and firewalled peer connectivity, smoother interface performance and a broad range of usability, protection and reliability improvements.

This page brings together the key information around the project: an overview of the client, technical guidance for advanced connectivity features, release notes for recent work, and support sections that help users get started and troubleshoot effectively.

Highlights

Modern Connectivity

Connect more peers through NAT Traversal, including "eServer assisted LowID to LowID" and "Kad firewalled to firewalled" transfers, with QUIC or uTP used for direct client to client traffic.

Cross-Platform Access with Docker

Use the official emuleai/emuleai Docker image on Windows, Linux, or macOS with the release launcher package. Xpra presents eMule AI as a separate desktop window while noVNC provides browser based VNaccess.

Built for Large Libraries

Background loading, virtual lists, shared file caching and lighter bulk operations keep startup, searches, transfers and very large libraries smooth and responsive.

Dark Mode & UI Polish

A complete dark theme, refined dialogs, clearer progress overlays and responsive layouts make everyday use cleaner and easier to follow.

Protection Panel

Centralized anti-leecher and abuse controls let you choose fair actions for suspicious behavior with timed rules, friend exemptions and safeguards against false positives.

Safer Downloads

Download Validator, Download Inspector, spam detection and blacklist tools help avoid duplicates, misleading files and unwanted results before they become a problem.

Smarter Upload Management

Flexible slot targets, idle slot recycling, adaptive buffering and queue controls help make better use of both everyday and high-bandwidth connections.

Shared Library Automation

Background scanning, folder watching, startup caches and automatic subdirectory sharing keep large collections current with less manual work.

Network Privacy & Control

VPN Guard, interface binding, IPv6 support, firewall rule management and optional port randomization provide clearer control over how eMule AI connects.

Better Search & Discovery

Configurable eD2K and Kad limits, fast filters, restored search tabs and clearer file states make large result sets easier to explore and manage.

Backup, Migration & Recovery

A guided migration wizard, automatic backups, safer state saving and restored source data help protect your setup and resume interrupted work quickly.

Everyday Productivity

Useful toolbar actions, richer sortable columns, client notes, flexible notifications and 116 built-in languages make common tasks faster and more accessible.

Connection
Capacities

Enter the real maximum speeds of your internet connection. Accurate upload capacity is especially important because eMule uses it to manage bandwidth without blocking normal connection traffic.

Set optional speed limits for eMule. Leaving a limit disabled allows eMule to use the available bandwidth while other connection controls remain active.

Enter the highest download speed your internet connection can provide. The value uses Mbit/s or KB/s, which matches the unit currently shown beside this field. This describes your connection capacity and does not limit eMule by itself.

Limit the download bandwidth eMule may use. Enable this when other applications need part of the downstream capacity or leave it disabled for no download limit.

Upload

Enter the highest upload speed your internet connection can provide. The value uses Mbit/s or KB/s, which matches the unit currently shown beside this field. Enter the real line rate, or zero to let automatic upload management estimate it.

Upload limit

Limit the upload bandwidth eMule may use. Keep some upload capacity free for acknowledgements and protocol traffic or download speed and web browsing may suffer.

Client Port

Configure the listening ports used for incoming peer connections. Router forwarding and firewall rules must match these values to obtain High ID and an open Kad status.

TCP port

Set the TCP listening port used for direct eD2K client connections and file transfers. TCP establishes a reliable connection before data is exchanged. Forward this port through the router and allow it through the firewall.

Test Ports

Open the eMule port test in your browser to check whether the configured TCP and UDP ports accept incoming connections. A failed test usually means the firewall or router settings need attention.

UDP

Set the UDP listening port used by Kad and by low overhead messages between clients and servers. UDP sends individual datagrams without a persistent connection. Forward this port for an open Kad status.

disable

Disable eMule UDP communication when this port cannot be used. Kad will be unavailable and more client communication must use TCP, which increases overhead.

Use UPnP to Setup Port Forwarding

Let eMule ask a compatible router to forward the configured ports automatically at startup. Use manual port forwarding if UPnP is unavailable or unreliable.

Open listen ports in Windows Firewall

Choose new TCP and UDP listening ports from the specified range each time eMule starts. Your firewall and router must allow the full range.

Random port range

Random port range start

Enter the first port that eMule may choose for randomized TCP and UDP listening ports.

Random port range end

Enter the last port that eMule may choose for randomized TCP and UDP listening ports. It must not be lower than the start value.

Set the maximum number of sources eMule keeps for each download. A very high value increases memory use and network requests without always improving speed.

Set the maximum number of simultaneous network connections eMule may open. Values beyond the operating system and router capacity can cause instability or connection failures.

Adjust NTFS daylight file time

On NTFS file systems, file timestamps may appear to shift by one hour if Daylight Saving Time (DST) is active. Enabling this option adjusts file times accordingly, which may help match files more accurately across DST changes.
WARNING: This adjustment may cause unnecessary rehashing of files when DST switches, potentially leading to performance issues, especially with a large number of shared files. To reduce this impact, consider enabling the "Allow DST time tolerance".

Allow DST time tolerance

Enabling this option allows a 1-hour tolerance when comparing file timestamps, which helps prevent unnecessary rehashing of files caused by Daylight Saving Time (DST) differences. This is especially useful when the "Adjust NTFS daylight file time" option is enabled, as it may shift file times by one hour depending on DST status. If this tolerance is not allowed, identical files may appear different and get rehashed each time DST switches, which can negatively impact performance, especially with large numbers of shared files.

Ratio1=Uploaded Total x 2 / Downloaded Total
Ratio2=SQRT(Uploaded Total+2 )

Start:1 min:0.1 max:100
Ratio:1:1.5
l-modifier=100*((1-1/(1+exp((3*{MB uploaded to us}^2-{MB downloaded from us}^2)/1000)))^6.6667)

TK4 Credit

If you are sharing completed files and someone downloads data from one of these from you, they probably cannot give you data back as you have all of the file. So in the TK4 system their credit rating remains unchanged. If you are downloading a file and someone takes data from you from the parts of the file you have they will be subject to the credit system and depending on how much they have given they may have their credit rating reduced. At any time if anyone gives you data they get a credit rating increase. The credit system works basically like this:
Start:10 Allowance:'Mb given'/4 + 1MbIf 'Mb taken' greater than 'Mb given + allowance + 1Mb (they have taken more than they gave + some)
Then if the file is a file we are downloading
If '(Mb taken - allowance)' is between 1 to 81Mb credit multiplier range = 9 to 1 math: 10 - square root('(Mb taken - allowance)' - 'Mb given') if '(Mb taken - allowance)' is greater than 81Mb credit multiplier range = 1 to 0.14 math: 9 / square root('(Mb taken - allowance)' -'Mb given')
If 'Mb given' is greater than 'Mb taken' (they have given more than they have taken)
Credit multiplier range = 10 to 342 math: 10 + log(2.72+('Mb given'-'Mb taken)*4)+('Mb taken/12)
There is no absolute range for multipliers but I have quoted a range of 0.14< to >342 (Assumes around a max. 4Gb difference) with a starting point of 10. Clients with failed SUI will start at 10, but not go up.

Docker End-User Guide

The official eMule AI Docker image runs the Windows x64 application under Wine in a Linux amd64 container. The release package provides launchers for Windows, Linux, and macOS. The launchers download or update the image, create the container, mount persistent data, publish the selected ports, wait for the display service, and open the user interface.

Downloads: Get eMuleAI_Docker_Package.zip from the eMule AI Releases page. The official image is emuleai/emuleai on Docker Hub.

5. What the Launcher Does

  1. Checks that Docker is installed and the Docker engine is running.
  2. Reads and validates every setting in eMuleAI.conf.
  3. Creates the selected persistent data directory.
  4. When IMAGE_SOURCE=registry, checks REGISTRY_IMAGE:REGISTRY_TAG on Docker Hub. If the local image is current, Docker reuses the existing layers. If a newer image exists, Docker downloads the changed layers and updates the local tag.
  5. Removes the existing container with the configured CONTAINER_NAME, then creates a replacement using the selected image. Persistent data is retained because it is stored outside the container.
  6. Waits until the container and selected display service are ready.
  7. When Xpra is selected and AUTO_ATTACH_XPRA=yes, starts the installed Xpra client automatically.
  8. When CLEAN_OLD_IMAGES=yes, removes older image tags and unused image IDs only after the replacement container starts successfully.
Updating eMule AI: Run the same operating-system script again. No separate docker pull command is needed in registry mode.

6. Open the Interface

Xpra Mode

With DISPLAY_MODE=xpra, the launcher connects to tcp://127.0.0.1:14500 by default and opens eMule AI as a separate desktop window. The port, username, password, client path, Windows backend, and scaling behavior are configurable.

noVNC Mode

With DISPLAY_MODE=novnc, open the URL printed by the launcher. With default settings it is:

http://127.0.0.1:6080/vnc.html?autoconnect=1&resize=scale

7. Stop and Start Later

  • Close eMule AI normally from its application window. With the default RESTART_POLICY=no, the container remains stopped.
  • Run the operating-system launcher again to start a fresh container using the same persistent data.
  • To stop it from a terminal, use docker stop eMuleAI when the default container name is unchanged.
  • Do not use the same data directory simultaneously with a native eMule AI instance or another container.
Ports, High ID, and Remote Access

The default eMule ports are 4662/TCP and 4672/UDP. For High ID, the values configured in eMuleAI.conf, the ports configured inside eMule AI, and the ports forwarded by the router should match:

EMULE_TCP_PORT=4662   -> router 4662/TCP
EMULE_UDP_PORT=4672   -> router 4672/UDP

The launcher maps each host port to the same container port. Avoid translating a different host port to the eMule port because eMule announces its configured listening port to peers.

DISPLAY_BIND_ADDRESS=127.0.0.1 keeps Xpra or noVNC local to the computer. Publishing a display service on 0.0.0.0 exposes it to other network interfaces and should only be done with appropriate network restrictions, authentication, and transport protection.

Persistent Data Layout

The selected operating-system data directory is mounted as /data in the container. It contains:

  • config: eMule AI configuration and state files.
  • Incoming: completed downloads.
  • Temp: incomplete downloads and part files.
  • logs: eMule AI logs.
  • wineprefix: persistent Wine environment.
  • xpra: Xpra authentication and related data.

Replacing or updating the container does not remove this directory. Back it up according to your normal data-protection policy.

Troubleshooting

First confirm that Docker is running and that the selected display port is not already used by another application. Useful commands with the default container name are:

docker ps -a --filter name=eMuleAI
docker logs eMuleAI
docker port eMuleAI
docker inspect eMuleAI

For Xpra problems, confirm that the host Xpra client is installed and that XPRA_CLIENT_EXE_WINDOWS, XPRA_CLIENT_EXE_LINUX, or XPRA_CLIENT_EXE_MACOS points to the correct executable when automatic detection does not find it.

For pointer alignment or moved-window click problems on Windows, keep XPRA_CLIENT_BACKEND_WINDOWS=gtk, XPRA_DESKTOP_SCALING=off, and ensure the Xpra canvas is at least as large as the complete Windows desktop.

For noVNC problems, verify the printed browser URL and the configured NOVNC_PORT.

Migration & Configuration Guide

eMule AI includes a built-in Migration Wizard. For most users, this should be the first and preferred way to restore an older eMule setup. Manual migration is still available for advanced users who want to copy and review files themselves.

Migration Wizard

Start here first. On first launch, eMule AI can automatically offer to restore an older eMule configuration after language selection. The wizard checks the default Windows profile location used by classic eMule and can also let you choose another folder manually.

  • Automatic first-run flow: If a legacy configuration is detected, the wizard can guide you through the restore process right away.
  • Manual later use: You can request it again from Tools > Migration Wizard. When started from the Tools menu, eMule AI schedules the wizard for the next application start, so you should restart the client.
  • What it restores: The wizard copies supported legacy configuration files into the local eMule AI config folder and shows a clear summary when the operation finishes, including any files that could not be copied.
  • Why it is recommended: It is the safest way to preserve your identity, credits, Kad state, server lists, shared configuration, and other compatible settings without manually sorting files.
Recommended workflow: Use the Migration Wizard first. Choose manual migration only if you deliberately want to inspect, copy, or exclude files yourself.

Manual Migration

Manual migration is still supported for advanced users. eMule AI stores its active configuration in a local config folder next to eMuleAI.exe, not in the Windows user profile location used by older eMule builds.

If you prefer to copy files yourself, close eMule AI first, copy only the files you actually want to keep, and then start the client again. If your older setup stores downloads.txt in the main eMule folder instead of inside config, remember to bring that file over as well.

While manual migration is supported, a clean configuration can still be the better choice if you want to start with eMule AI's modern defaults for current hardware, high-speed networks, and newer connection handling.

Important Files to Preserve

If you migrate manually, these files are the most valuable ones to keep because they preserve your identity, credits, and network state.

Identity & Reputation

Your reputation and queue ranking with other clients are tied to your unique identification. Without these files, you lose your "earned luck" and will start from zero in other users' queues.

Credits

What are these credits and why should I collect them?
Credits reward users who upload. The transferred amount of data determines the amount of credit you are given with a particular client. They are not global, meaning that they can only be used on the client who granted them to you. Credits are a major modifier when calculating how fast a client progresses through eMule's queue. The more credits you have the faster you will advance in other people's queues.
Where can I see my credit score?
To prevent manipulating the score, the credits are not stored locally on your hard disk but on the other client's hard disk. It is not possible to see your own credits.
I lost / deleted my clients.met file. Are my credits gone?
Losing this file doesn't mean that your credits are gone, but all credits other people have on your client are. eMule creates a backup of the clients.met file (clients.met.BAK) so you can restore the file by renaming the backup to clients.met.
Are credits secure?
eMule uses a key handshake method to make sure the credits belong to this special client. Credits stored for a user are only granted if this authentication has been successful.
How to lose the credits anyway?
As described above, the credits are coupled with the secure user authentication. The secret key for this authentication is calculated from information which is stored in preferences.dat and cryptkey.dat. Should one of these files be lost or deleted, you will no longer be able to prove that you are the rightful owner of the credits.
Note: If you lose only cryptkey.dat, it is mandatory to delete the preferences.dat file too.

High ID vs Low ID Explained

An ID is a value calculated from the client's IP address. It is assigned by the server when eMule successfully connects. The purpose of the ID is to indicate whether proper bidirectional communication is possible between two clients.

How to identify your status?

eMule displays your ID status graphically as colored arrows on the globe icon in the lower-right corner, next to the connected server. After connecting, the ID value itself is also visible in the Server's Log window.

High ID

A High ID means the port chosen in your connection settings (default 4662) is open and freely accessible from the internet. Your client is fully reachable by others.

Low ID

A Low ID means your connection port is blocked or cannot be reached. This is typically caused by firewalls, routers, or proxy servers. Any ID value less than 16,777,216 is considered a Low ID.

Disadvantages of a Low ID

  • Server Routing: Since your IP is not directly reachable, all requests (queue, connection, etc.) must be routed through the server you are connected to. This increases CPU load on the server.
  • Server Limitations: Many servers limit the number of Low ID users or may ban them entirely to reduce overhead.
  • Reduced Sources: Two clients with Low IDs cannot connect to each other because messages cannot be routed over two different servers. This leads to fewer available sources for your downloads.
  • Lost Messages: On busy servers, messages can get lost, causing you to miss important queue progression or download requests, which results in fewer credits and slower downloads.

Important Technical Notes:

  • Data Transfer: The ID only affects control message exchange. Actual data transfer is still handled client-to-client (Direct or via NAT-T paths in eMule AI).
  • No "Higher" ID: If your ID is High, it doesn't matter how high it is. There are no extra advantages to an "even higher" ID.
  • ID Calculation: To convert a High ID to an IP (A.B.C.D): High ID = A + 256*B + 256*256*C + 256*256*256*D.
  • Source Filtering: eMule drops some Low ID sources to minimize server load. Only sources on the same server or those downloading from you are accepted.

Troubleshooting Low ID

If you're stuck with a Low ID despite having correct settings, try connecting to a different server or using the Port Tester in connection options. Low user servers often have more resources to handle Low ID routing than congested ones.

NAT Traversal (NAT-T) & uTP

What it is

This feature combines NAT Traversal (NAT-T) and uTP to create direct, reliable peer connections even when both sides sit behind NAT or firewalls.

In eMule AI it unlocks LowID-to-LowID transfers with UDP hole punching and uTP, improving connectivity and stability on difficult networks.

Because the NAT-T/uTP signaling and relay flow are eMule AI-specific, both peers must be running eMule AI to benefit; the more eMule AI users online, the more often this path is available.

How it works

ED2K HighID / LowID
Assigned by ED2K servers while connected to a server. HighID means inbound TCP is reachable. LowID means inbound TCP is blocked by firewall/NAT.
Kad Open / Firewalled
Kad status is server-independent. Open + Verified means UDP is reachable. Firewalled means UDP callback/hole punching is limited or unavailable.
Kad Buddy
Classic Kad mechanism. Used for Kad callback/rendezvous signaling only (not file data). In eMule AI Kad LowID<->LowID transfers, uploader side needs this buddy.
eServer Buddy (New in eMule AI)
A HighID eMule AI client on the same eServer that relays signaling for LowID<->LowID. This is a client-side eMule AI feature and does not require server NAT extension support.
uTP
UDP-based transport used by NAT traversal for hole punching and adaptive congestion control.
Enabling uTP switches peer connections to UDP-based transport with adaptive congestion control, while NAT Traversal adds direct callback and UDP hole punching to connect peers behind NATs.

How it works - end-to-end

  • IDs & reachability: HighID = inbound TCP reachable; LowID = inbound TCP blocked. Kad Open means UDP callback is verified; Kad Firewalled means direct UDP callback is unavailable.
  • Data vs control: File data uses TCP or uTP. Kad lookups, publishes, pings and callbacks use UDP.
  • Direct UDP callback (LowID->HighID): HighID peer sends a UDP callback; LowID opens outbound TCP back. No buddy is involved.
  • Kad buddy path: For Kad LowID<->LowID, uploader side needs Kad buddy and both sides must exchange UDP hole-punch traffic.
  • eServer Buddy path (new in eMule AI): LowID requester uses a connected HighID buddy on the same eServer. This relay is client-driven and does not rely on server NAT extension opcodes.
  • When target is not connected: Buddy can ask the server with classic OP_CALLBACKREQUEST, then continue relay after target comes back.
  • Compatibility: Requester side needs eServer buddy support. Target side can still be contacted through the relay flow.
  • Timers: Buddy discovery runs every 60 seconds (while server-LowID). Keep-alive ping runs every 20 seconds.
  • Diagnostics: Check logs for [eServerBuddy], OP_ESERVER_RELAY_REQUEST, OP_ESERVER_PEER_INFO, OP_HOLEPUNCH, and uTP.

Kad Buddy vs eServer Buddy (eMule AI)

  • Kad Buddy: Kad network callback/rendezvous helper. Used in Kad LowID<->LowID flow.
  • eServer Buddy: eMule AI specific relay over same eServer. This is the newer path added by eMule AI.
  • Main difference: Kad buddy depends on Kad path; eServer buddy depends on same-server eD2K path.
  • Common point: Both relay only control signaling, never the actual file data.

Buddy specifics

  • Who qualifies (eServer Buddy): HighID peer, connected TCP socket, same eServer as you, and advertised slot availability.
  • Slot model: Serving capacity is configurable between 3 and 100 peers (default 3).
  • Concurrency: Buddies carry control signaling only; file payload remains end-to-end between peers.
  • Security & abuse control: Buddy requests are rate-limited (max 3 requests per 10-minute window per peer) and stale relay contexts are expired automatically.

Diagrams

LowID <-> LowID via Kad rendezvous. In eMule AI, only the uploader-side buddy is required for this path; downloader-side buddy is optional.
LowID <-> LowID via eServer Buddy relay. No Lugdunum NAT extension is required; buddy may optionally use classic OP_CALLBACKREQUEST to bring target online.

Setup & tips

Make UDP reachable

  • Open/forward the configured UDP port on your router (or enable UPnP).
  • Allow the app in OS firewall (UDP + TCP outbound).
  • Verify Kad shows Open and the UDP firewall test is verified.

Enable NAT-T / uTP

  • Ensure NAT Traversal (uTP) is enabled in Preferences.
  • Keep both peers on recent versions (supports rendezvous & uTP).

Buddy stability

  • For Kad LowID<->LowID, keep uploader-side serving buddy active.
  • For eServer Buddy path, keep your HighID buddy connected and on the same eServer as the target source.
  • eMule AI sends keep-alive pings and renews buddy discovery automatically while LowID.

Troubleshooting

LowID <-> LowID fails
Kad path: verify uploader has serving buddy and both sides can exchange UDP hole-punch traffic. eServer Buddy path: verify same-server condition and requester-side connected eServer buddy.

Use the peer's UDP port (not TCP) for bootstrap and confirm firewall/router rules.

FAQ

Is ED2K LowID the same as Kad Firewalled?
No. ED2K HighID/LowID exists only while connected to a server and reflects TCP reachability. Kad Open/Firewalled is Kad's status about UDP reachability, independent of servers.
Does NAT-T help if one peer is Kad Firewalled?
No. Direct callback and hole punching require UDP reachability (Kad Open + Verified) on both sides.
What is the difference between Kad Buddy and eServer Buddy?
Kad Buddy is the classic Kad callback helper. eServer Buddy is a newer eMule AI feature for same-server LowID<->LowID relay. They are different paths with different prerequisites.
Can LowID <-> LowID work with only one buddy?
Yes. In Kad rendezvous, uploader-side buddy is required and downloader-side buddy is optional. In eServer Buddy path, requester-side eServer buddy is required while target-side buddy is not.
What about CGNAT and symmetric NAT?
Those NATs often create short-lived or address-dependent mappings which may break hole punching. Results vary and can be unreliable.

Report an issue

The most helpful reports include a clear reproduction path, the exact build you used, and diagnostic files when available.

Open the GitHub issue tracker

To report a bug, document a reproducible problem, or suggest an improvement, please use the eMule AI GitHub Issues page.

What to include in a good report

  • The exact eMule AI version or release name.
  • Your Windows version and whether you are using x86 or x64 build.
  • Clear step-by-step instructions that reproduce the problem.
  • What you expected to happen and what actually happened instead.
  • Relevant screenshots, log excerpts, or test files if they help explain the issue.

Some problems can be understood from a normal description and a screenshot, but crashes, unexpected assertions, and difficult network issues usually need stronger diagnostics. The three sections below explain the most useful reporting methods for those cases.

General advice: try to keep the reproduction small and focused. A short test with clean logs and matching files from the same run is usually much more useful than a very large mixed report from a long session.

Verbose logs for diagnostics

If the issue does not crash the client but still needs deeper analysis, verbose logs are usually the best starting point. They are especially helpful for connection behavior, NAT-T and uTP issues, unexpected state changes, retries, queue logic, and similar problems that are hard to explain with screenshots alone.

How to enable the most detailed logging

  1. Delete or move any older files from the local logs folder first, so the new logs are easier to identify.
  2. Open Options > Extended.
  3. Enable Verbose.
  4. Set Log Level to 5. This is the highest and most detailed log level.
  5. Enable Log NAT traversal events if the problem is related to connectivity, NAT-T, or uTP.
  6. Keep the feature you are testing enabled while reproducing the problem. For example, if you are reporting a NAT-T or uTP issue, make sure Enable uTP & NAT Traversal (NAT-T) remains enabled during the test.
  7. Reproduce the problem with the simplest possible test case you can find.
  8. After the test, compress the newly created files from the logs folder and attach them to the GitHub issue, or upload them to a file host if they are too large for GitHub.
Tip: if possible, reproduce the issue with one focused test case instead of a long general session. Short, clean verbose logs are much easier to analyze.

Crash reports and dump files

If eMule AI crashes, shows an invalid argument message, or closes unexpectedly, please attach a crash dump whenever possible. A dump usually lets the developers identify the failing function much faster than a screenshot or Event Viewer entry alone.

Note: A normal Release build is enough to create a crash dump. You do not need a special Debug build for reporting crashes.

Built-in crash dump method

  1. Open Options > eMule AI > Advanced Preferences.
  2. Enable Create crash dump.
  3. Close eMule AI completely and start it again.
  4. Reproduce the crash.
  5. If a dialog asks whether to create a diagnostic file, click Yes.
  6. Open the local config folder of your eMule AI installation and attach the newest .dmp file to the GitHub issue.
Important: The dump is typically written to the local config folder, not necessarily the main program folder.

Fallback method with ProcDump

If no dump is created by the built-in method, you can use ProcDump from Microsoft Sysinternals as a fallback.

procdump.exe -accepteula -ma -e -t -w eMuleAI.exe C:\Dumps

After running that command, start eMule AI normally, reproduce the crash, and then attach the newest dump file from C:\Dumps.

Before you attach the dump

  • Compress the dump with ZIP or 7z if it is large.
  • If GitHub does not accept the file because of its size, upload it to a file host and share the download link in the issue.
  • Include the exact crash message, the last action you performed, and whether the problem is reproducible.

Network, NAT-T, and uTP issue reports

If your problem looks network-related, for example sources stay at Unknown, many sources end up as Too many connections, NAT-T or uTP works only on some networks, or transfers improve after disabling Enable uTP & NAT Traversal (NAT-T), please include a deeper report than a normal bug description.

Why this matters: for this class of issue, screenshots alone are usually not enough. The most useful reports combine a simple reproduction path, full verbose logs, and a short packet capture taken during the same test session.

What to include in a network-related report

  • The exact eMule AI version or release name.
  • Your Windows version and whether you are using the x86 or x64 build.
  • Your TCP port, UDP port, whether UPnP is enabled, and whether you are using manual port forwarding.
  • Your current network setup, for example direct connection, home router, VPN, virtual machine, or mobile hotspot.
  • A short description of what you expected and what happened instead.
  • The smallest reliable reproduction case you can find, ideally one file with many sources.
  • Whether the problem changes when Enable uTP & NAT Traversal (NAT-T) is disabled.
  • Whether the same test behaves differently on another network, for example when you connect through your phone's hotspot instead of your usual router.

Capture the network side with Wireshark

Wireshark is a free network analysis tool. It helps show whether the expected UDP and uTP packets are actually reaching your PC during the failing test. You can download it here: https://www.wireshark.org/download.html.

  1. Download and install Wireshark.
  2. Open Wireshark and start capturing on the network adapter your PC is actively using, usually Ethernet or Wi-Fi.
  3. In the filter box, enter a filter for the UDP port used by eMule AI. Example: udp.port == 4672. Replace 4672 with your actual UDP port.
  4. Start the capture before reproducing the problem.
  5. Reproduce the issue in eMule AI while the verbose logging options from the section above are enabled.
  6. Let the capture run for about 30 to 60 seconds while the problem is actively happening.
  7. Stop the capture and save it as a .pcapng file.
  8. Compress the .pcapng file and attach it to the same GitHub issue together with the matching verbose logs from the same test session.
Important: the packet capture and the verbose logs should come from the same test run whenever possible. That makes it much easier to compare what happened on the network side with what eMule AI reported internally at the same moment.

Before you submit the report

  • Compress large logs and packet captures with ZIP or 7z.
  • If GitHub rejects the files because of their size, upload them to a trusted file host and share the download links in the issue.
  • Please mention if the problem disappears on another network, after changing the UDP port, or after disabling NAT-T manually.
  • Remember that packet captures can contain network metadata such as IP addresses, so review them before sharing if needed.
